# Spec: Sufficiency Meta-Analysis Block

## Overview

The sufficiency meta-analysis answers the question: **given a target ensemble of N
structures, what fraction must be collected experimentally, and what fraction can be
generated by the model?**

It is a **pure plotting block**. It computes nothing. It reads per-run analysis
NPZ files that already exist on disk from completed pipeline runs and produces
figures that visualize how generative quality evolves across values of
`max_data`. If data is missing for some conditions it warns and plots what is
available. If no data is available at all it prints a message and skips the block.

---

## 1. Config Schema

### Global config additions

`max_data` is a field under `data` (`data.max_data`). It accepts either a single integer
or a list of integers. When a list is provided, the pipeline schedules one full run
per value. All other global config fields (including `training_split` and `seed`) apply
unchanged to every run in the sweep.

```yaml
# Existing global fields (unchanged)
data:
  split_seed: 0
  training_split: 0.8
  # New data field.
  # Single integer or list of integers.
  # null = use all available structures.
  # Each value in a list defines a separate full pipeline run.
  max_data: [100, 250, 500, 1000, 2500, 5000]

meta_analysis:
  sufficiency:
    enabled: true
    overwrite_existing: false
    save_pdf_copy: true   # save figures as PDF in addition to PNG
```

That is the complete meta-analysis config. No metrics list, no
parallelization options, no statistical test parameters, no curve fitting options.

### What the sweep implies

With the config above, the pipeline runs once per `max_data` value.

- If `data.training_split` is a single float (e.g. `0.8`), then `training_split` is
  fixed and the meta-analysis heatmaps degenerate to a single row.
- If `data.training_split` is a list (e.g. `[0.6, 0.8]`), then the pipeline schedules
  one full run per `(max_data, training_split)` combination, and the
  sufficiency meta-analysis includes those combos in the per-seed heatmap.

## 2. max_data Implementation

`max_data` is consumed by `load_data()` in `src/utils.py`. When set, structures are
subsampled using the global seed so that a given `(max_data, seed)` pair always
produces the same subset.

```python
def load_data(path: str, max_data: int | None = None, seed: int = 0) -> np.ndarray:
    coords = np.load(path)["coords"]
    if max_data is not None and max_data < len(coords):
        rng = np.random.default_rng(seed)
        idx = rng.choice(len(coords), size=max_data, replace=False)
        coords = coords[idx]
    return coords
```

When `max_data` is a list, the orchestrator in `run.py` iterates over the values and
dispatches one full pipeline run per value (or per `(max_data, training_split)`
combo when `training_split` is a list). All other config fields apply unchanged per
run.

**Validation at config load time:**

- All `max_data` values must be positive integers
- Any value exceeding the total number of structures in the dataset emits a warning
and is clamped to the dataset size, since dataset size is not known until load time

---

## 3. Data Sources

The meta-analysis reads per-run analysis NPZ files produced by the normal pipeline
analysis blocks (`analysis.rmsd_gen` and `analysis.q_gen`). It does not invoke any
model, run any inference, or recompute any metric.

Required files per run (variance=1 generation), written by the analysis blocks
when their `save_data: true` (or temporarily retained if scoring is enabled; see
purging):

```
{output_dir}/distmap/{seed}/euclideanizer/{eu_seed}/analysis/
    rmsd/gen/{run_name}/data/rmsd_data.npz
    q/gen/{run_name}/data/q_data.npz
```

Each file is expected to contain:

- `rmsd_data.npz`:
  - `gen_to_test`: 1D float32 array of per-structure min-RMSD values
    (one value per test structure)
- `q_data.npz`:
  - `gen_to_test`: 1D float32 array of per-structure max-Q values
    (one value per test structure)

Metrics are fixed: `min_rmsd` and `max_q`. These are not configurable.

**If a required NPZ file is missing for a given `(max_data, training_split)` combo:**

- Print a warning identifying the `(max_data, training_split)` combo and the missing file path
- Continue with remaining conditions
- The missing combo is omitted from the figure, and the figure title notes which combos were missing

**If no files are found across any condition:**

- Print: `"Sufficiency meta-analysis: no analysis NPZ data found. Skipping."`
- Exit the block without producing any output

**Note on prerequisites:** The analysis blocks must have written `rmsd_data.npz` /
`q_data.npz` for the generation variance used by scoring (`variance=1`), or the
pipeline must have temporarily retained those NPZ files until meta-analysis
completes. When `scoring.enabled` is false, these NPZ files are only present if
the corresponding analysis blocks have `save_data: true` (because the pipeline
does not write NPZ data otherwise). The meta-analysis does not validate this at
config load time but prints a warning at runtime if it finds no data.

## 4. Data Purging

Purging always runs and is governed entirely by each block's own `save_data`
parameter, consistent with the rest of the pipeline.

However, when `meta_analysis.sufficiency.enabled` is true, the pipeline must
(for now) **defer deletion of only the necessary analysis NPZ files** until after
the sufficiency meta-analysis has consumed them.

### Required deletion deferral (necessary-file purge override)
The sufficiency meta-analysis needs only:

- `analysis/rmsd/gen/*/data/rmsd_data.npz` (for the generation run where variance=1)
- `analysis/q/gen/*/data/q_data.npz` (for the generation run where variance=1)

When scoring is enabled, the pipeline performs “save-then-delete after scoring” so
scoring can read these NPZ files. The sufficiency meta-analysis adds one more
consumer step; therefore the pipeline must not delete the NPZ *required by the
meta-analysis* until after that step has run.

After the sufficiency meta-analysis completes, purging proceeds according to the
usual `save_data` flags of those analysis blocks:

- If an analysis block has `save_data: false`, the pipeline deletes the NPZ data
  directories after meta-analysis finishes.
- If an analysis block has `save_data: true`, the pipeline keeps them.

## 5. Output Directory Structure

```
{output_dir}/
  meta_analysis/
    sufficiency/
      seed_{seed}/
        distributions/                       # one 2-panel figure per max_data
          max_data_{max_data}/
            distributions_rmsd_q.png
            distributions_rmsd_q.pdf         # only if save_pdf_copy: true

          sufficiency_heatmap_rmsd_q.png
          sufficiency_heatmap_rmsd_q.pdf      # only if save_pdf_copy: true
```

## 6. Distribution Figures

For each seed directory and each `max_data` condition that has the required analysis
NPZ files for **both** metrics, the block produces a **single combined figure** with
**two columns** (min-RMSD | max-Q) and **one row per `training_split`** (largest split
at the top). Each row uses **borderless filled** density histograms
(`HIST_BINS_DEFAULT` bins; edges from the 1st–99th percentile pooled over splits for
that `max_data`). Row color encodes `training_split` via viridis on `[0, 1]`; a
percentage label appears in the top-left of the left panel for that row. Y limits are
matched within each column so rows are comparable.

Saved as:

- `meta_analysis/sufficiency/seed_{seed}/distributions/max_data_{max_data}/distributions_rmsd_q.png`
  (and `.pdf` if `save_pdf_copy: true`)

### Layout

**Columns:**

- Left: histogram of per-test-structure `min_rmsd` (`gen_to_test` from
  `analysis/rmsd/.../rmsd_data.npz`); x label on the bottom row only.
- Right: histogram of per-test-structure `max_q` (`gen_to_test` from
  `analysis/q/.../q_data.npz`); Q axis range clipped to `[0, 1]` where needed.

**Titles (top row):** `"Min RMSD | Max Data={max_data}"` and `"Max Q | Max Data={max_data}"`.

**Training-split colorbar:**

- Horizontal strip **below** the bottom row of panels, **~75%** of the combined axes
  width, centered; label **“Training Split”** **above** the strip; ticks **0%** and
  **100%** (colormap spans the full viridis range).

**Missing-condition behavior:**

- If one metric is missing for a specific `(max_data, training_split)` combo,
  omit that combo from both panels and print a warning naming the missing NPZ
  file(s).
- If no `(max_data, training_split)` combos have both metrics for a given `max_data`,
  skip the combined distribution figure for that `max_data`.

---

## 7. Heatmap Figures

For each seed directory, the block produces a **single combined 2-panel heatmap**
figure summarizing medians across the `(max_data, training_split)` grid:

- Left panel heatmap: median `min_rmsd` for each `(max_data, training_split)`
  combo (using the `gen_to_test` arrays from `rmsd_data.npz`)
- Right panel heatmap: median `max_q` for each `(max_data, training_split)`
  combo (using the `gen_to_test` arrays from `q_data.npz`)

The right heatmap uses the **median of max-Q values** for each grid cell (not the
maximum max-Q).

Saved as:

- `meta_analysis/sufficiency/seed_{seed}/heatmap/sufficiency_heatmap_rmsd_q.png`
  (and `.pdf` if `save_pdf_copy: true`)

### Layout

- X axis label on **each** panel: **“Max Structures”** (tick labels are the numeric
  `max_data` values, ordered).
- Y axis: `training_split` (percentages, low at bottom, high at top).
- **`imshow` aspect:** `equal` so cells are square; figure width scales with grid size
  (`plot_config.SUFFICIENCY_HEATMAP_CELL_IN`).

**Shared colorbar + viridis normalization:**

- One horizontal colorbar below both panels; viridis on **`0` … `1`**.
- **Per-panel normalization:** within each heatmap, finite medians are linearly
  mapped to `[0, 1]` using that panel’s min and max (masked cells excluded), so the
  two panels are comparable in *relative* terms. Colorbar label: **“Normalized Median”**
  (no extra caption on the figure).

Panel titles:

- Left: `"Median Min RMSD"`
- Right: `"Median Max Q"`

## 8. Dashboard Integration

The pipeline dashboard includes a **Meta-Analysis** view (View → **Meta-Analysis**):

- **Full-width layout** per seed: median heatmap (`sufficiency_heatmap_rmsd_q.png`)
  and, when present, **all** `distributions/max_data_*/distributions_rmsd_q.png`
  figures in a vertical stack (Title Case section labels in the UI).
- Assets are copied into `dashboard/assets/` like other dashboard figures; the
  manifest carries a per-seed **`sufficiency_meta`** object with `heatmap` and
  `distributions` paths.

**Seed detail** view still lists the sufficiency heatmap as a normal block (and
aggregated child-run blocks) when the user opens that seed from Browse.

If no sufficiency outputs exist for any seed, the Meta-Analysis view shows an
empty state instead of placeholder panels.

## 9. Validation Summary


| Condition | Behaviour |
| --- | --- |
| `max_data` value exceeds dataset size | Warning at load time, clamped to dataset size |
| Required RMSD/Q analysis NPZ missing for some `(max_data, training_split)` | Warning at runtime naming the missing file(s); omit the combo |
| `analysis.rmsd_gen.save_data` / `analysis.q_gen.save_data` false and `scoring.enabled=false` | Warning at runtime noting likely cause of missing NPZ files |
| Required RMSD/Q analysis NPZ missing for all `training_split` combos for a given `max_data` | Message printed and the combined distribution figure for that `max_data` is skipped |
| Required RMSD/Q analysis NPZ missing for all `max_data` for a given seed | Message printed and the sufficiency block is skipped entirely for that seed |
